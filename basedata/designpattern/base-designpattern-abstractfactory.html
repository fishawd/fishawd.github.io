# 模式定义

抽象工厂模式(Abstract Factory)提供一个创建一系列相关或相互依赖对象的接口，而无需指定他们的具体类。抽象工厂内的方法不是任意堆砌的，而是一系列相关或相互依赖的方法。 抽象工厂模式用来创建多个等级的产品，而工厂方法模式只生产一个等级的产品。

例如，假如工厂方法能生产CPU，无论是Intel还是AMD，他们都属于同一个产品等级。 这个时候又需要生产主板和内存条，那么主板和内存条也分别属于不同的产品等级。

CPU、主板、内存是三个不同的产品等级，产品等级由相同的产品组成，例如CPU等级由AMD的CPU和Intel的CPU组成一个产品等级。 CPU、主板、内存这三个相互关联的产品组成一个产品家族。 例如AMD的CPU、主板、内存组成一个产品家族， Intel的CPU、主板、内存组成另一个家族。

# 模式结构

  • 抽象产品(AbstractProduct): CPU，主板，内存
  • 具体产品(Product)：不同品牌的CPU，主板，内存
  • 抽象工厂(AbstractFactory)
  • 具体工厂(ConcreteFactory)

# 优点缺点

优点：

  • 客户端无须知道具体创建过程，降低了耦合
  • 可以轻易地通过更换工厂更换产品族
  • 增加新的产品族无需修改原来的系统，只需要增加新的类即可

缺点：

  • 增加新的产品等级需要修改工厂，从而所有实现了抽象工厂的类都需要修改，带来很大的不便

# 使用时机

  • 系统有多个产品族
  • 系统一次只使用一个产品族， 即同一产品族的产品是一起使用的
  • 系统不依赖于产品的实现
